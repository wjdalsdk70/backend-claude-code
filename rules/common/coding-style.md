# Coding Style

## 불변성 (CRITICAL)

항상 새 객체를 생성하고, 기존 객체를 변경하지 않는다.

```
// 의사코드
잘못: modify(original, field, value) → original을 직접 변경
올바름: update(original, field, value) → 변경된 새 복사본 반환
```

이유: 불변 데이터는 숨겨진 부작용을 방지하고 디버깅을 용이하게 하며 안전한 동시성을 가능하게 한다.

## 핵심 원칙

### KISS (단순하게 유지)

- 실제로 작동하는 가장 단순한 해결책 선호
- 성급한 최적화 지양
- 영리함보다 명확성 우선

### DRY (반복하지 않기)

- 반복되는 로직은 공유 함수나 유틸리티로 추출
- 복사-붙여넣기 구현 드리프트 방지
- 반복이 실제로 발생할 때만 추상화 도입

### YAGNI (필요할 때만)

- 필요하기 전에 기능이나 추상화 구축 금지
- 투기적 일반화 지양
- 단순하게 시작하고 압박이 실제로 있을 때 리팩토링

## 파일 구성

많은 작은 파일 > 적은 큰 파일:
- 높은 응집도, 낮은 결합도
- 일반적으로 200-400줄, 최대 800줄
- 대형 모듈에서 유틸리티 추출
- 타입이 아닌 기능/도메인으로 구성

## 오류 처리

항상 오류를 포괄적으로 처리:
- 모든 레이어에서 명시적으로 오류 처리
- UI 코드에서는 사용자 친화적 오류 메시지 제공
- 서버 측에서 상세 오류 컨텍스트 로깅
- 오류를 조용히 삼키지 않기

## 입력 검증

항상 시스템 경계에서 검증:
- 처리 전 모든 사용자 입력 검증
- 가능한 경우 스키마 기반 검증 사용
- 명확한 오류 메시지와 함께 빠르게 실패
- 외부 데이터를 신뢰하지 않기 (API 응답, 사용자 입력, 파일 내용)

## 코드 품질 체크리스트

작업을 완료로 표시하기 전:
- [ ] 코드가 읽기 쉽고 명명이 적절함
- [ ] 함수가 작음 (<50줄)
- [ ] 파일이 집중됨 (<800줄)
- [ ] 깊은 중첩 없음 (>4 레벨)
- [ ] 적절한 오류 처리
- [ ] 하드코딩된 값 없음 (상수 또는 설정 사용)
- [ ] 불변 패턴 사용됨
